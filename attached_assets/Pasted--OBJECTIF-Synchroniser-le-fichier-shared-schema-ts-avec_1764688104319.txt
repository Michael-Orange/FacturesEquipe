MAPPING DES TYPES SQL → DRIZZLE
Type SQL	Type Drizzle	Exemple
VARCHAR	varchar("column_name")	userName: varchar("user_name")
INTEGER	integer("column_name")	categoryId: integer("category_id")
SERIAL	serial("column_name")	id: serial("id")
BOOLEAN	boolean("column_name")	hasTva: boolean("has_tva")
TIMESTAMP	timestamp("column_name")	createdAt: timestamp("created_at")
NUMERIC	numeric("column_name")	Ou varchar si stocké en string
TEXT	text("column_name")	description: text("description")
Notes :
Si nullable → Ne pas ajouter .notNull()
Si default → Ajouter .default(valeur)
Si foreign key → Ajouter .references(() => autreTable.colonne)
CONTRAINTES ET DEFAULTS
Exemples :
// NOT NULL
userName: varchar("user_name").notNull(),

// DEFAULT
archive: boolean("archive").default(false),

// DEFAULT avec SQL
id: varchar("id").default(sql`gen_random_uuid()`),

// FOREIGN KEY
supplierId: varchar("supplier_id").references(() => suppliers.id),

// FOREIGN KEY avec CASCADE DELETE
invoiceId: varchar("invoice_id").references(() => invoices.id, { onDelete: "cascade" }),

// UNIQUE
token: varchar("token").notNull().unique(),
POINTS D'ATTENTION
NE PAS modifier la structure BDD
Ce brief synchronise le schema TypeScript avec la BDD existante
Aucune migration SQL à créer
Pas de drizzle-kit push
Vérifier les noms de colonnes
SQL utilise snake_case : user_name
TypeScript utilise camelCase : userName
Drizzle gère la conversion automatiquement
Types VARCHAR pour les montants
Les montants sont stockés en VARCHAR dans cette app (pas NUMERIC)
Garder varchar("amount_ttc") même si ce sont des nombres
Colonnes optionnelles
Si is_nullable = YES en BDD → Ne pas ajouter .notNull()
Exemple : description: varchar("description") (pas .notNull())
Relations entre tables
Ajouter les .references() pour les foreign keys
Respecter les CASCADE DELETE si définis en BDD
Ne pas casser l'existant
Les colonnes déjà présentes dans schema.ts sont correctes
Ne modifier QUE pour ajouter les colonnes manquantes
Ne pas supprimer de colonnes existantes

## ❓ QUESTIONS DE VALIDATION

**Après implémentation, vérifie :**

1. **Toutes les colonnes sont-elles présentes ?**

2. **Les types sont-ils corrects ?**
   - VARCHAR → `varchar()`
   - INTEGER → `integer()`
   - BOOLEAN → `boolean()`
   - TIMESTAMP → `timestamp()`

3. **Les contraintes sont-elles respectées ?**
   - NOT NULL en BDD → `.notNull()` dans schema
   - DEFAULT en BDD → `.default()` dans schema
   - FOREIGN KEY en BDD → `.references()` dans schema
