# OBJECTIF
Implémenter un système complet de gestion des paiements multiples pour les factures

# CONTEXTE
Actuellement, l'application ne gère pas les paiements partiels. Les factures peuvent être payées en plusieurs fois (acomptes successifs).
Besoin de tracker chaque paiement avec son montant, sa date et son type de règlement.

EXEMPLE D'USAGE :
- Facture de 100 000 F
- Paiement 1 : 30 000 F le 01/12 par Virement
- Paiement 2 : 40 000 F le 15/12 par Espèces
- Paiement 3 : 30 000 F le 30/12 par Mobile Money
- Statut : Soldée ✅

# ARCHITECTURE BASE DE DONNÉES

## 1. Modifications table `invoices`

**Nouvelle colonne :**
```sql
ALTER TABLE invoices ADD COLUMN payment_status VARCHAR DEFAULT 'unpaid';
payment_status: varchar
Valeurs possibles : "unpaid" | "partial" | "paid"
Défaut : "unpaid"
Logique :
"unpaid" : Aucun paiement enregistré
"partial" : Au moins 1 paiement mais total < montant TTC
"paid" : Total des paiements = montant TTC (facture soldée)
2. Nouvelle table payments
Création :
CREATE TABLE payments (
  id SERIAL PRIMARY KEY,
  invoice_id INTEGER NOT NULL REFERENCES invoices(id) ON DELETE CASCADE,
  amount_paid VARCHAR NOT NULL,
  payment_date VARCHAR NOT NULL,
  payment_type VARCHAR NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  created_by INTEGER REFERENCES users(id)
);
Schéma Drizzle (shared/schema.ts) :
export const payments = pgTable("payments", {
  id: serial("id").primaryKey(),
  invoiceId: integer("invoice_id").notNull().references(() => invoices.id, { onDelete: "cascade" }),
  amountPaid: varchar("amount_paid").notNull(),
  paymentDate: varchar("payment_date").notNull(),
  paymentType: varchar("payment_type").notNull(),
  createdAt: timestamp("created_at").defaultNow(),
  createdBy: integer("created_by").references(() => users.id)
});

export const insertPaymentSchema = createInsertSchema(payments).pick({
  invoiceId: true,
  amountPaid: true,
  paymentDate: true,
  paymentType: true,
  createdBy: true
});
Contraintes métier :
amountPaid : Doit être > 0
paymentDate : Format YYYY-MM-DD
paymentType : Valeurs existantes (Virement, Espèces, Carte bancaire, Mobile Money, etc.)
createdBy : User ID de la personne qui enregistre le paiement

SCHÉMA DE VALIDATION - Exemple backend
Validation ajout paiement (POST /api/payments)
// Pseudo-code de validation
function validateAddPayment(invoiceId, amountPaid, paymentDate, userId) {
  // 1. Facture existe ?
  const invoice = await db.getInvoice(invoiceId);
  if (!invoice) throw Error("Facture introuvable");
  
  // 2. User autorisé ?
  if (invoice.createdBy !== userId) throw Error("Non autorisé");
  
  // 3. Facture déjà soldée ?
  if (invoice.paymentStatus === "paid") throw Error("Facture déjà soldée");
  
  // 4. Calcul reste à payer
  const totalPaid = await db.getTotalPaid(invoiceId);
  const remaining = parseFloat(invoice.amountTTC) - totalPaid;
  
  // 5. Montant valide ?
  const amount = parseFloat(amountPaid);
  if (amount <= 0) throw Error("Montant invalide");
  if (amount > remaining) throw Error(`Ne peut pas dépasser ${remaining} F`);
  
  // 6. Date valide ?
  if (paymentDate < invoice.createdAt) throw Error("Date antérieure à la facture");
  if (paymentDate > today) throw Error("Date dans le futur");
  
  // OK, créer le paiement
}
